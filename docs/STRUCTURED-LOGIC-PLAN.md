# Structured Logic Layer — Implementation Plan

> Public-safe plan for the structured-logic-extraction layer in ptcg-deck-builder.
> Solves the PTCG card-text regional ambiguity (e.g. TW「70以下」= ≤70 in Taiwan but <70 in Hong Kong).

## Problem

PTCG card text has region-dependent semantics. The 繁中 (TW) official text uses「以下」/「以上」whose
boundary inclusiveness varies by region (Taiwan: 以下 includes the boundary = ≤; Hong Kong: excludes = <).
Relying on TW free-text embeddings cannot reliably disambiguate this. Many cards carry such thresholds
(好友寶芬 HP「70」以下, 緊急滑板 HP「30」以下, 等級球 HP「90」以下, 紫竽 獎賞卡 3 張以下).

A second gap: situational evaluation (博士的研究 discards the whole hand; 土龍節節 needs evolution to be
online) is currently done by crude keyword matching, not structured attributes.

## Solution core

Extract structured predicates from the **Japanese canonical text** (`jp_cards`), because JP wording is
unambiguous:

- `以下` = ≤ (inclusive), `未満` = < (exclusive), `以上` = ≥, `超過` = >, `ちょうど` = ==

**Design principle — LLM as parser, not oracle:**
The LLM only parses JP text into structure + quotes a `jp_source_span` + reports `raw_token`/`raw_value`.
The comparison **operator is never decided by the LLM** — it is produced by a deterministic token→op
dictionary applied by a pure-code verifier that also checks the span literally exists in the source text.
Same anti-hallucination spirit as the existing `assistant._strip_unverified_examples`.

## Predicate taxonomy

`hp_threshold{op,value,applies_to}` · `count_threshold{op,value,dim}`
(dim = prize|hand|bench|energy|damage_counters|deck) · `draw` · `discard` · `search_deck` · `switch`
· `search_trash` · `heal` · `attach_energy` · `place_damage_counters` · `evolve` · `condition`

One extraction feeds both gaps: `hp_threshold` kills the「以下」ambiguity (Gap A);
`discard`/`draw`/`evolve` derive role axes for situational eval (Gap B).

## Token → operator dictionary

```
{"以下":"<=", "未満":"<", "以上":">=", "超過":">", "ちょうど":"=="}
```

## Verified data-model facts

- **TW↔JP link = `set_code + set_number` JOIN** (the `services/tcgdex/bridge.py` module uses exactly this;
  JP and TW share `set_code` for SV-series 2023+). Standard-legal regulation marks H/I/J yield ~3347
  joinable cards.
  ```sql
  SELECT c.card_id tw, j.card_id jp, j.description jp_desc
  FROM cards c JOIN jp_cards j
    ON c.set_code = j.set_code
   AND split_part(c.set_number,'/',1) ~ '^[0-9]+$'
   AND split_part(j.set_number,'/',1) ~ '^[0-9]+$'
   AND split_part(c.set_number,'/',1)::int = split_part(j.set_number,'/',1)::int;
  ```
- Not link fields (empty/dirty, don't use): `cards.japanese_name`, `cards.jp_id`,
  `jp_cards.chinese_name`, `pokedex_number` (dirty; a bare pokedex join explodes to ~97k false matches).
- `processed_cards` table exists but is **empty** (schema: `card_id` PK, `card_name`, `original_text`,
  `logic_json` text, `status`, `attempts`, `last_updated`).
- `cards.ai_logic_json` column does **not** exist in the live schema, yet
  `services/crawler/update_pokemon.py` (~line 310) inserts `ai_logic_json='{}'` and
  `services/deck_importer/deck_importer.py` (~line 386) reads `card_data['ai_logic_json']` — that write
  path is currently broken. Decision: centralize structured logic in `processed_cards` (single source of
  truth) and redirect these two sites.
- `services/ai_assistant/` (indexer/tools/assistant) references `logic_json`/`processed_cards`
  **zero times** — the structured-logic layer is fully disconnected from retrieval today.
  - `indexer._card_doc` (~79-115) embeds only description+skills, no logic.
  - `indexer._fetch_card_docs` (~118-132) does not JOIN `processed_cards`. `STANDARD_MARKS=("H","I","J")`.
  - `tools.semantic_search_cards` (~196), `tools._card_payload`, `tools._analyze_deck_play_plan`
    (keyword-based).

## Eval methodology & gate

Golden set (~30-50 cards, stratified easy/medium/hard) hand-annotated by a PTCG-knowledgeable human
(the ground truth; cannot be auto-generated by an LLM or it just replicates LLM bias). The eval runner
computes, per category and difficulty:

- **threshold-precision** ≥ 95% (op+value+dimension exact match; the「以下」line)
- **op-accuracy** ~100% (since op is deterministic from the token map, anything below 100% is a bug,
  not a tuning knob)
- **action-F1** ≥ 80%
- **hallucination-rate** < 2% (fabricated predicates with no grounded span)
- **span-grounding-rate** ~100%

Ship the full backfill only after the golden gate passes.

### Current Phase 1 gate status

Phase 1 currently uses a deterministic, rule-based extractor rather than an LLM parser. Against the expanded
17-card golden set (9 seed + 8 adversarial long-tail cases), `python3 backend/eval/runner.py --use-extractor`
reports:

- overall threshold-precision: `1.000`
- overall op-accuracy: `1.000`
- overall action-F1: `0.833`
- overall hallucination-rate: `0.000`
- overall span-grounding-rate: `1.000`

Interpretation: the extractor is precise and safe for the Gap A threshold ambiguity, but it is not yet a
complete action/effect extractor. Adversarial cases intentionally expose missing Gap B coverage: `switch`,
`heal`, `attach_energy`, `evolve`, `search_trash`, discard costs, and broader deck-search patterns.

## Landing plan (phases)

- **Phase 0** — schema + golden set + eval runner. Migration SQL (`golden_set` table;
  `processed_cards` ALTER add `predicates jsonb`/`confidence`/`version`; reconcile `ai_logic_json`).
  Seed golden set. Eval runner. *(done locally & committed; migration not yet applied to production —
  applied via the deploy process)*
- **Phase 1** — extractor `services/logic_extractor/` (`schema.py`/`extractor.py`/`verifier.py`). Core is
  a pure function `(card_text) -> list[predicate]`; DB read/write is a thin adapter. Dev against a local
  fixture (golden cards' JP text exported via read-only SELECT); dev writes nothing to the production DB.
- **Phase 2** — deploy + Gap A-only backfill + ingest hook. Adapter reads `jp_cards` → extractor core →
  writes only threshold/search-threshold predicates that are safe for the「以下」ambiguity. Store metadata/version
  such as `gap_a_threshold_only` so downstream code does not treat missing action predicates as proof that a
  card has no action effect. Admin endpoint to trigger backfill. Hook into `jp_crawler.save_card_to_db`.
  Adapter entrypoint: `services.logic_extractor.adapter.backfill_gap_a_threshold_only(...)`; it defaults to
  `dry_run=True` and requires `001_logic_layer.sql` before any write path is used. Admin endpoints:
  `GET /api/admin/logic-extractor/gap-a/status` and `POST /api/admin/logic-extractor/gap-a/backfill`.
- **Phase 3 (Gap A payoff, MVP end)** — wire retrieval: `_fetch_card_docs` LEFT JOIN `processed_cards`;
  `_card_doc` appends normalized predicate string + metadata; `_card_payload` exposes predicates;
  `semantic_search_cards` gains `predicate_filter`; re-embed.
  Retrieval wiring should tolerate pre-migration deployments by returning empty predicates when the logic columns
  are not present yet. Current reliable filter scope is `gap_a_threshold_only`, especially `hp_threshold` and
  threshold-backed `search_deck` predicates.
- **Phase 6** — admin annotation GUI (golden-set authoring + review queue), writes DB live.
- **Phase 4 (Gap B)** — derive role axes (discard_cost/needs_evolution/burst_vs_steady/per_turn_limit)
  from predicates; rewrite `_analyze_deck_play_plan`.
- **Phase 5** — agent grounding (SYSTEM_PROMPT prefers predicates; rule-critical claims must cite a
  predicate/span; extend `_strip_unverified_examples`).
- **Phase 7 (optional)** — CI/CD.

## Phase 1 first steps (generic)

1. Confirm dev env: clone on `main`, clean; production DB readable read-only for JP text.
2. Read-only SELECT the seed cards' JP canonical via the set_code+set_number join above.
3. Save as `backend/eval/fixtures/golden_cards.json` (local dev fixture; do not write to production DB).
4. Draft `backend/migrations/001_logic_layer.sql`.
5. Hand-write gold predicates for seed cards (per taxonomy + token dictionary).
6. Scaffold `backend/eval/runner.py`.

## Work discipline

- Small steps, report each; don't run one huge tool round that stalls.
- Verify by read-only SELECT — don't assume a column is populated (several link fields here are empty).
- All code changes happen in the clone and are pushed via git; migrations are code (committed) and
  applied to the production DB through the deploy process, never by automation writing directly to
  production.
- The Grep tool fails on alternation patterns (`a|b`); use single terms.
